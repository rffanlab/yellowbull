# 任务执行引擎 — 详细设计

> 模块职责：接收拆解后的 Step 列表，按依赖序执行每步，处理失败、分支、循环、障碍排除，最终判定任务结果。

---

## 一、核心参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 单步超时 | 120 秒 | 单步执行最长等待 |
| 任务总超时 | 1800 秒 | 30 分钟 |
| 总步数预算 | 100 步 | 含所有子任务步骤 |
| 单步重试次数 | 2 次 | 可重试错误（网络/超时） |
| 子任务最大步骤 | 3 步 | 单个子任务的上限 |
| 嵌套深度 | 不限制 | 靠预算/超时/重复检测防护 |
| 补救轮数 | 2 轮 | 完成判定后最多补救 2 次 |

## 二、执行流程全景

│  ├─ 预校验: 拓扑排序、孤立步骤、循环依赖                       │
│  └─ 失败 → 任务直接失败                                      │
└────────────────────┬────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────────┐
│  MAIN LOOP: 获取下一步                                       │
│  ├─ 过滤: pending + depends_on 满足 + 未被分支跳过            │
│  ├─ 排序: 关键优先 → 读优先 → 拓扑序                          │
│  └─ 无下一步 → 进入完成判定                                  │

                     │
                     ▼

                     │
                     ▼

│  判断步骤类型                                                │
│  ├─ 普通步骤 → 直接执行                                     │

                     │
                     ▼

                     │
                     ▼

│  │                                                           │

## 三、普通步骤执行

```
execute_step(step):

  1. 前置检查
     - 检查 depends_on 是否全部 done/skipped
     - 有 failed 依赖 → 级联跳过，标记 skipped

  2. 收集输入数据
     - 从 context_store 按 input_from 取数据
     - 按 input_format 做格式校验

  3. 工具路由
     - 根据 tool_hint 匹配工具
     - 无匹配 → 触发动态工具生成

  4. 构建执行 Prompt
     - step.description
     - 输入数据
     - 工具说明
     - expected_output 约束

  5. 调用工具执行 (timeout=120s)
     - 捕获异常

  6. 结果处理
     - 空结果 → 视为失败
     - 错误码 → 进入重试/失败处理
     - 成功 → 存入 context_store
```

## 四、条件分支步骤执行

```
execute_branch_step(branch_step):

  1. 正常执行该步骤（如"检查服务状态"）

  2. 拿到结果后，交给 LLM 评估 branch_condition
     Prompt:
       "以下条件是否满足？
        条件: {branch_condition}
        依据: {step_result}
        请回答 true 或 false。"

  3. 根据评估结果:
     - true  → true_next 标记为 pending，false_next 标记为 skipped
     - false → false_next 标记为 pending，true_next 标记为 skipped

  4. 后续"获取下一步"时，skipped 步骤会被过滤

边界情况:
  - true_next / false_next 为空 → 该分支末端，不再产生新步骤
  - true_next 未声明 depends_on 分支点 → 自动补全依赖关系
  - 分支条件评估失败（LLM 无法判断）→ 默认走 false 分支，记录警告
```

## 五、循环迭代步骤执行

```
execute_loop_step(loop_step):

  1. 从 context_store 取 loop_input_step 的结果

  2. 校验结果是否为可迭代集合
     - 列表/数组 → 继续
     - 单个值 → 包装为单元素列表
     - 空集合 → 整个循环标记 skipped，输出空列表
     - 不可迭代 → 标记 failed，记录错误

  3. 设置循环保护参数
     - max_iterations: 受全局 step_budget 约束
     - timeout_per_iteration: 120 秒
     - timeout_total: 受全局任务超时约束

  4. 遍历集合元素
     FOR item IN collection:

       a. 替换 description 中的变量
          "检查 {file} 的 import" → "检查 a.py 的 import"

       b. 构建独立执行上下文
          - 注入当前 item 数据
          - 继承上游步骤的 context

       c. 执行该步骤
          - 同普通步骤执行流程

       d. 迭代失败处理 (见第六章)

       e. 检查全局保护
          - 预算耗尽 → 终止循环
          - 超时 → 终止循环

  5. 将累积结果存入 context_store

边界情况:
  - 循环集合元素过多 → 达到预算/超时后截断
  - 循环数据源步骤 failed → 循环直接标记 failed
  - 循环步骤本身也是分支点 → 不允许，校验阶段拦截
  - 循环嵌套循环 → 不允许，校验阶段拦截
  - 循环中某步需要用户确认 → 暂停循环，等待响应后恢复
```

## 六、失败处理与智能障碍排除

### 6.1 失败处理流程

```
step_failed(step, error):

  1. 判断是否可重试
     - 网络错误 → 可重试
     - 超时 → 可重试
     - 工具内部错误 → 可重试
     - 逻辑错误（如文件不存在）→ 不可重试

  2. 可重试
     - retry_count < 2 → 退避后重试
     - retry_count >= 2 → 进入下一步

  3. 不可重试 / 重试耗尽 → 交给 LLM 分析
     Prompt:
       "步骤 '{step.description}' 执行失败。
        错误: {error}
        已重试 {retry_count} 次。
        请分析:
        a. 这是可恢复的错误吗？
        b. 需要做什么才能排除这个障碍？
        请返回处理策略: retry / resolve / skip / abort"

  4. LLM 返回策略

     策略 A — retry (直接重试)
       → 重试该步骤

     策略 B — resolve (需排除障碍)
       → 进入障碍排除流程 (6.2)

     策略 C — skip (跳过)
       → 标记 failed，继续后续步骤

     策略 D — abort (终止)
       → 标记 failed，中断整个任务

  5. 级联影响
     - 该步骤最终 failed → 依赖它的全部步骤标记 skipped
     - 递归传播直到无更多受影响步骤
```

### 6.2 障碍排除流程

```
resolve_obstacle(step, error):

  1. LLM 生成障碍排除方案
     Prompt:
       "步骤 '{step.description}' 因 '{error}' 失败。
        请给出排除障碍的子任务:
        - 障碍描述
        - 排除步骤（最多 3 步）"

  2. 创建子任务 (SubTask)
     - 目标: 排除障碍
     - 继承父任务上下文（只读）
     - 步骤列表: LLM 生成的步骤

  3. 暂停当前流程
     - 将当前执行上下文压入执行栈
     - 标记为 paused

  4. 执行子任务
     - 走正常执行引擎流程
     - 子任务自身也可触发障碍排除（嵌套）

  5. 子任务完成
     - 成功 → 恢复原流程，重试原步骤
     - 失败 → 跳过原步骤，继续

  6. 重复障碍检测
     - 记录已处理的障碍签名: hash(step_id + error_type)
     - 同一障碍反复出现 2 次 → 不再尝试排除
     - 例: 安装 mypy 失败 → 重试安装又失败 → 跳过
```

### 6.3 子任务数据结构

```python
SubTask {
  id: str
  parent_task_id: str              # 父任务
  parent_step_id: str              # 触发该子任务的步骤
  goal: str                        # 排除障碍的目标
  obstacle_description: str        # 障碍描述
  steps: list[Step]                # 子任务自己的步骤列表
  status: "pending" | "running" | "done" | "failed"
  context_inherit: bool            # 是否继承父任务上下文
  max_steps: 3                     # 子任务最多 3 步
}
```

## 七、步骤选择策略

```
get_next_step(pending_steps):

  过滤条件:
    1. 排除 status != pending 的
    2. 排除 depends_on 未满足的
    3. 排除被分支跳过的（skipped）

  排序优先级:
    1. 关键步骤优先 (is_critical=true)
    2. 读操作优先于写操作 (工具类型排序)
    3. 拓扑序靠前的优先

  返回: 最高优先级步骤 或 None（无可执行步骤）

边界情况:
  - 所有 pending 步骤的 depends_on 都不满足
    → 说明存在 failed 依赖 → 级联标记 skipped
    → 若级联后仍无进展 → 任务卡死，进入兜底
```

## 八、执行栈设计

```
执行模型: 栈式执行

  ExecutionStack:
    - current: Task/SubTask       # 当前执行的任务
    - paused: list[TaskContext]   # 暂停的任务上下文
    - results: dict[str, any]     # 全局结果存储

  压栈场景:
    1. 原步骤失败 → 需要排除障碍
       压入 SubTask
       原任务标记 paused

    2. 子任务完成
       弹出 SubTask
       恢复原任务

  示例:
    用户任务: "检查所有 .py 文件的 import"
      ↓
    执行中: step_2 (循环检查文件)
      ↓
    迭代到 "utils.py" 时失败: 缺少 mypy 依赖
      ↓
    压栈: SubTask (安装 mypy)
    暂停: 主任务 (停在 utils.py 迭代处)
      ↓
    SubTask 完成: 安装成功
      ↓
    恢复: 主任务，重试 utils.py
      ↓
    继续: 下一个文件迭代
```

## 九、无限嵌套防护

```
虽然不限制嵌套层数，但设置以下防护:

  1. 全局步数预算
     - 每个任务 max_total_steps=100
     - 包含所有子任务的步骤
     - 耗尽 → 强制终止，进入完成判定

  2. 全局超时
     - 任务总超时 30 分钟
     - 到达 → 强制终止

  3. 重复障碍检测
     - 记录已处理的障碍签名
     - 同一障碍反复出现 2 次 → 不再尝试排除

  4. 子任务步骤上限
     - 单个子任务最多 3 步
     - 防止子任务本身过于复杂

  5. LLM 递归感知
     - Prompt 中注入当前嵌套深度
     - "当前嵌套深度: N，请尽量给出简洁的排除方案"
```

## 十、任务完成判定

```
evaluate_task_completion():

  统计:
    critical_failed = failed 步骤中 is_critical=true 的数量
    non_critical_failed = failed 步骤中 is_critical=false 的数量
    done_count = done 步骤数
    total = 总步骤数

  判定:
    1. critical_failed > 0 → 任务失败
    2. 全部 done → 完全成功
    3. 非关键 failed 且 完成率 >= 80% → 部分成功（良好）
    4. 非关键 failed 且 完成率 < 80% → 补救判定

  补救判定 (最多 2 轮):
    交给 LLM:
      "已完成 {done_count}/{total} 步
       失败 {non_critical_failed} 步（均为非关键）
       目标: {goal}
       成功标准: {success_criteria}
       请判断:
       a. 已完成的部分是否足够满足目标？
       b. 若不够，能否通过补救达成？
       c. 补救需要哪些步骤？（最多 3 步）"

    LLM 返回:
      a. 足够 → 部分成功，生成报告
      b. 可补救 → 创建补救子任务，执行
      c. 不可补救 → 任务失败，生成报告

  报告内容:
    - 任务目标
    - 执行结果（成功/部分成功/失败）
    - 完成的步骤列表
    - 失败的步骤及原因
    - 子任务执行记录（如有）
    - 耗时统计
    - 后续建议
```

## 十一、超时与中断

```
超时控制:
  - 单步超时: 120 秒
  - 任务总超时: 1800 秒（30 分钟）

中断处理:
  - 用户主动取消:
    → 停止当前执行
    → 保存已完成结果
    → 生成中断报告
    → 任务状态设为 "cancelled"

  - 超时中断:
    → 停止当前执行
    → 保存已完成结果
    → 判断已完成步骤是否"够用"
    → 够用 → 部分成功
    → 不够 → 任务失败

  - 预算耗尽:
    → 同超时中断处理

  - 系统异常（崩溃/断电）:
    → 状态已持久化
    → 重启后可从断点恢复（后续版本）
```

## 十二、状态机设计

### 12.1 步骤状态流转

```
  pending ──→ running ──→ done
    │           │
    │           │→ failed
    │           │
    ↓           ↓
  skipped     (retry)
                │
                ↓
              running

  状态转换规则:
    pending → running:  执行引擎选中该步骤
    pending → skipped:  依赖失败 或 分支跳过
    running → done:     执行成功
    running → failed:   执行失败且重试耗尽
    failed → running:   重试（仅可重试错误）
```

### 12.2 任务状态流转

```
  pending ──→ running ──→ done
    │           │
    │           │→ failed
    │           │
    ↓           ↓
  cancelled   cancelled
```

### 12.3 执行栈状态

```
  idle ──→ executing ──→ paused
    │           │            │
    │           │            ↓
    │           │       executing (子任务)
    │           │            │
    │           │            ↓
    └───────────┴──────→ resumed
                         │
                         ↓
                     executing
```

## 十三、上下文传递规则

```
ContextStore 设计:

  命名空间隔离:
    - 主任务: context_store["task:{task_id}:{step_id}"]
    - 子任务: context_store["subtask:{subtask_id}:{step_id}"]
    - 子任务可读取父任务的 context（只读）

  数据格式转换:
    - 步骤声明了 input_format / output_format
    - 执行引擎在传递时做格式校验
    - 格式不匹配 → 尝试自动转换
    - 转换失败 → 步骤标记 failed

  示例:
    Step 1 output_format: file_list
    Step 2 input_format: file_path
    → 引擎从 Step 1 结果中取第一个文件路径传给 Step 2

    Step 1 output_format: text
    Step 2 input_format: json
    → 引擎尝试解析 text 为 JSON
    → 解析失败 → Step 2 标记 failed
```

## 十四、关键边界处理清单

| 边界场景 | 处理方式 |
|----------|----------|
| 依赖步骤 failed | 级联标记 skipped |
| 分支条件评估失败 | 默认走 false 分支 |
| 循环集合为空 | 循环标记 skipped |
| 循环元素过多 | 达到预算/超时后截断 |
| 循环迭代失败 | LLM 分析 → 重试/排除障碍/跳过/终止 |
| 子任务也失败 | 跳过原步骤，继续 |
| 同一障碍反复出现 | 记录签名，出现 2 次不再尝试 |
| 预算耗尽 | 强制终止，进入完成判定 |
| 超时 | 强制终止，进入完成判定 |
| 用户取消 | 保存已完成结果，生成中断报告 |
| 格式不匹配 | 尝试自动转换，失败则步骤 failed |
| 动态工具生成失败 | 标记 needs_user_help |
| 补救 2 轮仍失败 | 任务失败，报告给用户 |
| 嵌套过深 | 靠预算/超时/重复检测自然限制 |
| 所有 pending 都阻塞 | 级联 skipped，任务卡死进入兜底 |
| 循环步骤也是分支点 | 不允许，校验阶段拦截 |
| 嵌套循环 | 不允许，校验阶段拦截 |

## 十五、MVP 阶段决策

| 设计点 | 决策 |
|-------|------|
| 步骤并行执行 | MVP 暂不实现，按序执行 |
| 分支条件评估 | 全部交给 LLM 判断 |
| 障碍排除确认 | 自动执行，无需用户确认 |
| 嵌套深度 | 不限制，靠防护机制控制 |
| 断点恢复 | 预留接口，MVP 不实现 |
| 格式自动转换 | MVP 仅支持基础类型转换 |
