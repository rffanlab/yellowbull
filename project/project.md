# 老黄牛（YellowBull）核心设计文档

## 一、整体架构

```
┌─────────────────────────────────────────────────────┐
│                    CLI / UI 层                        │
│              (任务输入、结果展示、交互)                   │
├─────────────────────────────────────────────────────┤
│                  Agent 核心层                          │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐              │
│  │ 任务接收  │→│ 步骤拆解  │→│ 执行循环  │              │
│  └──────────┘ └──────────┘ └────┬─────┘              │
│                                 │                     │
│                    ┌────────────┼────────────┐        │
│                    ▼            ▼            ▼        │
│              ┌──────────┐ ┌──────────┐ ┌──────────┐  │
│              │ 工具调用  │ │ 结果检查  │ │ 修正循环  │  │
│              └──────────┘ └──────────┘ └──────────┘  │
├─────────────────────────────────────────────────────┤
│                  支撑层                               │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐              │
│  │ 工具系统  │ │ 记忆系统  │ │ 经验库    │              │
│  └──────────┘ └──────────┘ └──────────┘              │
├─────────────────────────────────────────────────────┤
│                  基础设施层                            │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐              │
│  │ LLM 接口  │ │ 配置管理  │ │ 持久化    │              │
│  └──────────┘ └──────────┘ └──────────┘              │
└─────────────────────────────────────────────────────┘
```

## 二、核心主流程

```
用户输入任务
    │
    ▼
┌──────────────┐
│  1. 任务接收  │  ← 解析用户自然语言，明确任务目标
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 2. 经验回顾  │  ← 查经验库：类似任务有没有成功/失败记录？
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 3. 步骤拆解  │  ← 将任务拆成有序的子步骤
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 4. 执行循环  │  ← 逐步骤执行（核心循环）
│  ┌────────┐  │
│  │ 调工具 │  │
│  └───┬────┘  │
│      ▼      │
│  ┌────────┐  │
│  │ 看结果 │  │
│  └───┬────┘  │
│      ▼      │
│  ┌────────┐  │
│  │ 需修正? │──YES→ 回到调工具（重试/换策略）
│  └───┬────┘  │
│      │NO    │
│      ▼      │
│  下一步骤    │
└──────┬───────┘
       │ 全部完成
       ▼
┌──────────────┐
│ 5. 结果汇总  │  ← 汇总所有步骤结果，输出最终答案
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 6. 记录经验  │  ← 将本次执行的关键信息存入经验库
└──────────────┘
```

## 三、每个子流程详细设计

### 3.1 任务接收

```
输入: 用户自然语言描述
输出: 结构化 Task 对象
```

#### 3.1.1 用户输入形式

| 输入类型 | 示例 | 特点 |
|---------|------|------|
| 一句话任务 | "帮我把项目里的日志级别改成 debug" | 简短，意图明确 |
| 多轮对话任务 | "我想重构用户模块" → "重点改认证部分" | 需要上下文累积 |
| 复杂任务 | "分析这个接口的性能瓶颈并优化" | 需要多步骤、多工具 |
| 模糊任务 | "看看代码有什么问题" | 需要追问澄清 |
| 附带上下文 | "参考 docs/api.md，生成对应的客户端代码" | 有文件/路径引用 |

#### 3.1.2 接收流程

```
用户输入
    │
    ▼
┌─────────────────┐
│ 预处理           │
│ - 去噪/格式化     │
│ - 检测是否附带文件 │
│ - 检测是否追问    │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 意图判断         │
│ - 是明确任务?     │──NO──→ 进入澄清流程
│ - 是对话/闲聊?    │
│ - 是配置指令?     │
└────────┬────────┘
         │ 明确任务
         ▼
┌─────────────────┐
│ LLM 结构化解析    │
│ 提取 Task 对象    │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 置信度评估        │
│ - 高置信 → 继续   │
│ - 低置信 → 追问澄清 │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 用户确认          │
│ - 简单任务自动执行  │
│ - 危险操作需确认    │  ← 文件删除、Shell 危险命令等
│ - 复杂任务需确认    │  ← 步骤超过 N 步的任务展示计划
└────────┬────────┘
         │
         ▼
    结构化 Task
```

#### 3.1.3 追问澄清机制

```
澄清维度:
  1. 范围不明确 → "你想重构整个模块还是某个函数？"
  2. 目标不明确 → "优化是指性能、可读性还是其他？"
  3. 约束不明确 → "有没有限制不能改接口？"
  4. 上下文缺失 → "你说的'那个接口'具体是哪个？"

追问规则:
  - 最多追问 3 轮，超过则给出默认假设并告知用户
  - 每次只问 1-2 个问题，避免信息过载
  - ★ 提供选项让用户快速选择（用户往往不知道需要澄清什么）
    例: "请确认操作范围: [1] 整个项目  [2] src/ 目录  [3] 仅当前文件"

MVP 阶段: MVP 不做任务类型分类，直接解析目标+约束+成功标准
```

#### 3.1.4 LLM 解析 Prompt 设计

```
System Prompt 结构:
  角色设定 → 你是任务解析专家
  任务说明 → 从用户输入中提取结构化信息
  输出格式 → JSON Schema 约束
  示例     → Few-shot 示范
  置信度规则 → 何时标记低置信

输出 Schema:
{
  "goal": "清晰的任务目标描述",
  "constraints": ["约束1", "约束2"],
  "success_criteria": ["标准1"],
  "context_files": ["path1"],
  "confidence": 0.0 ~ 1.0,
  "clarification_needed": "需澄清的问题（若无则为空）",
  "clarification_options": ["选项1", "选项2"]   # 供用户选择
}
```

### 3.2 经验回顾

```
输入: 结构化 Task 对象
输出: 相关经验列表（可能为空）

MVP 阶段决策: 代码结构预留经验库接口，MVP 阶段返回空列表
```

#### 3.2.1 未来完整流程

```
子流程:
  1. 从经验库检索相似任务
     - 按任务描述相似度匹配
     - 按工具使用模式匹配
     - 按失败模式匹配
  2. 若有成功经验 → 提取可复用策略
  3. 若有失败经验 → 提取避坑指南
  4. 将经验作为提示注入后续步骤
```

### 3.3 步骤拆解

```
输入: 结构化 Task 对象 + 经验提示
输出: 有序步骤列表 [Step, Step, ...]
```

#### 3.3.1 拆解策略

```
拆解原则:
  1. 原子性 — 每步只做一件事，便于独立评估和重试
  2. 有序性 — 有依赖关系的步骤按序排列
  3. 可回退 — 每步失败不影响全局
  4. 工具对齐 — 每步应能映射到一个具体工具

拆解粒度:
  - 太粗 → 一步包含多个操作，失败难以定位
  - 太细 → 步骤过多，LLM 调用成本高
  - 适中 → 一般 3-8 步，最多不超过 max_steps
```

#### 3.3.2 拆解流程

```
结构化 Task + 经验提示
        │
        ▼
┌─────────────────┐
│ 经验注入          │
│ 若有相似成功经验   │
│ → 参考其步骤模式   │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ LLM 拆解任务      │
│ 生成 Step 列表    │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 依赖关系校验      │
│ - 检查循环依赖     │
│ - 补全缺失依赖     │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 工具匹配          │
│ 每步绑定推荐工具   │
│ 若无匹配工具 → 标记 │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 步骤数校验        │
│ 超限 → 合并简单步骤 │
└────────┬────────┘
         │
         ▼
    最终 Step 列表
```

#### 3.3.3 LLM 拆解 Prompt 设计

```
System Prompt 结构:
  角色设定 → 你是任务规划专家
  可用工具清单 → 列出所有工具及描述（让 LLM 知道能力边界）
  经验提示   → "过往类似任务的经验: ..."
  拆解规则   → 原子性、有序性、工具对齐
  输出格式   → JSON Schema 约束

输出 Schema:
{
  "steps": [
    {
      "step_id": "step_1",
      "description": "步骤描述",
      "tool_hint": "file | shell | code | search",
      "depends_on": [],
      "expected_output": "期望结果",
      "is_critical": true/false   # 关键步骤失败则任务失败
    }
  ],
  "reasoning": "拆解思路说明"
}
```

#### 3.3.4 拆解示例

**输入任务**: "帮我把项目里的日志级别改成 debug，然后重启服务看看有没有报错"

**拆解结果**:
```
Step 1: 搜索项目中所有日志配置文件
  tool_hint: file (搜索)
  depends_on: []
  is_critical: true

Step 2: 读取找到的配置文件内容
  tool_hint: file (读取)
  depends_on: [step_1]
  is_critical: true

Step 3: 修改日志级别为 debug
  tool_hint: file (写入)
  depends_on: [step_2]
  is_critical: true

Step 4: 查找服务启动脚本/命令
  tool_hint: file (搜索)
  depends_on: [step_1]    ← 与 Step 2/3 无依赖，可并行
  is_critical: true

Step 5: 重启服务
  tool_hint: shell
  depends_on: [step_3, step_4]
  is_critical: true

Step 6: 检查服务日志是否有报错
  tool_hint: file (读取) + shell
  depends_on: [step_5]
  is_critical: false      ← 失败不影响"改配置"这个目标
```

### 3.4 执行循环（核心）

```
输入: 步骤列表
输出: 每步执行结果

对每个步骤:
  │
  ▼
┌────────────────────────┐
│ A. 选择工具             │
│   - 根据 step.tool_hint │
│   - LLM 决策最合适的工具  │
│   - 确定工具参数          │
└──────────┬─────────────┘
           │
           ▼
┌────────────────────────┐
│ B. 执行工具             │
│   - 调用工具函数          │
│   - 捕获输出/错误         │
│   - 记录执行日志          │
└──────────┬─────────────┘
           │
           ▼
┌────────────────────────┐
│ C. 结果评估             │
│   - 是否符合 expected?  │
│   - 是否有错误/异常？    │
│   - 质量是否达标？        │
└──────────┬─────────────┘
           │
      ┌────┴────┐
      │         │
   成功       失败
      │         │
      │    ┌────┴────┐
      │    │   重试?  │────YES → 回到 A（调整参数/换工具）
      │    └────┬────┘
      │         │
      │      已达最大重试
      │         │
      │    ┌────┴────┐
      │    │  跳过/   │
      │    │  降级?   │
      │    └────┬────┘
      │         │
      └────┬────┘
           ▼
      记录本步结果
      进入下一步
```

### 3.5 结果汇总

```
输入: 所有步骤的执行结果
输出: 最终报告

子流程:
  1. 收集所有步骤结果
  2. 标记成功/失败/跳过的步骤
  3. LLM 综合所有结果生成最终报告
  4. 若关键步骤失败 → 报告整体失败 + 原因
  5. 输出格式:
     - 任务完成情况
     - 每步结果摘要
     - 产出物（文件、代码等）位置
```

### 3.6 记录经验

```
输入: 完整执行记录
输出: 经验入库

子流程:
  1. 提取关键信息:
     - 任务类型
     - 使用的工具链
     - 成功/失败模式
     - 耗时
     - 关键决策点
  2. 经验评分（基于结果质量）
  3. 写入 SQLite 经验库
  4. 构建索引（便于后续检索）
```

## 四、工具系统设计

```
Tool (抽象基类)
  │
  ├── FileTool        # 文件读写、目录操作
  ├── ShellTool       # Shell 命令执行
  ├── SearchTool      # 网页搜索
  ├── CodeTool        # Python 代码执行
  └── CustomTool      # 用户自定义工具（扩展点）

每个工具具备:
  - name: 工具名
  - description: 描述（供 LLM 理解何时使用）
  - parameters: 参数 schema
  - execute(): 执行方法
  - validate(): 输入校验
```

## 五、数据结构设计

```python
# 核心数据流
Task {
  id: str
  goal: str
  constraints: list[str]
  success_criteria: list[str]
  context: str
  status: "pending" | "running" | "done" | "failed"
}

Step {
  id: str
  task_id: str
  description: str
  tool_hint: str
  depends_on: list[str]
  expected_output: str
  status: "pending" | "running" | "done" | "failed" | "skipped"
}

StepResult {
  step_id: str
  tool_used: str
  output: str
  error: str | None
  success: bool
  retry_count: int
  duration_ms: int
}

Experience {
  id: str
  task_type: str
  tool_chain: list[str]       # 使用的工具序列
  pattern: str                # 成功/失败模式描述
  lesson: str                 # 经验教训
  score: float                # -1.0 ~ 1.0
  created_at: datetime
}
```

## 六、状态机

```
Task 状态:
  pending → running → done
                ↘ failed

Step 状态:
  pending → running → done
                ↘ failed → (retry → running)
                ↘ skipped
```

## 七、设计决策记录

### 已确认

| 决策点 | 结论 | 说明 |
|--------|------|------|
| 任务分类 | MVP 不分类 | 直接解析目标+约束+成功标准 |
| 经验回顾 | MVP 预留接口 | 代码结构预留，返回空列表 |
| 用户确认机制 | 危险操作+复杂任务需确认 | 简单任务自动执行 |
| 澄清方式 | 提供选项让用户选择 | 用户往往不知道需要澄清什么 |

### 待确认

1. **LLM 选择策略** — 步骤拆解和工具调用，是用同一个模型还是不同模型？（复杂推理用大模型，简单调用用小模型可以省钱）

2. **经验库检索方式** — 简单关键词匹配 vs 向量相似度检索？（向量检索更准但需要额外依赖）

3. **代码执行安全** — 代码工具是否需要沙箱隔离？还是直接在本机执行？

4. **Shell 安全列表** — 哪些命令应该被禁止？（如 rm -rf, format 等）

5. **任务并行** — 无依赖的步骤是否需要支持并行执行？
