# 任务结果汇总 — 详细设计

> 模块职责：收集执行数据、评估任务达成度、生成用户报告、记录经验、等待用户反馈。

---

## 一、模块定位

```
执行引擎结束
    │
    ▼
┌──────────────────────────┐
│  结果汇总模块             │
│  1. 收集所有执行数据       │
│  2. 评估任务达成度         │
│  3. 生成用户报告           │
│  4. 记录经验（供未来学习）  │
│  5. 等待用户反馈           │
└──────────────────────────┘
```

---

## 二、输入数据

```
结果汇总模块能拿到什么:

  1. Task 原始定义
     - goal
     - success_criteria
     - constraints
     - confidence_score

  2. 步骤执行记录
     - 每个 step 的 id, description, status
     - 执行结果 (context_store 中的数据)
     - 耗时
     - 失败原因
     - 重试次数

  3. 子任务执行记录
     - 子任务触发原因（障碍描述）
     - 子任务步骤及结果
     - 子任务状态
     - 嵌套关系树

  4. 执行元数据
     - 开始时间 / 结束时间
     - 终止原因 (正常完成 / 超时 / 预算耗尽 / 用户取消)
     - 总步数消耗
     - 分支决策记录
     - 循环迭代统计

  5. 用户交互记录
     - 确认请求及响应
     - 追问及回答
     - 中途干预
```

## 三、结果评估流程

```
┌─────────────────────────────────────────────┐
│ 第 1 层: 机械统计                            │
│ - done / failed / skipped 数量               │
│ - 关键步骤是否全部完成                        │
│ - 完成率 = done / total                      │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────┐
│ 第 2 层: 规则判定                            │
│ - 关键步骤有 failed → 任务失败                │
│ - 全部 done → 完全成功                       │
│ - 完成率 >= 80% → 良好                       │
│ - 完成率 < 80% → 需要 LLM 评估               │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────┐
│ 第 3 层: LLM 综合评估                        │
│ 输入: 所有执行数据 + success_criteria         │
│ 输出:                                       │
│   - 最终结论 (成功/部分成功/失败)             │
│   - 达成度评分 (0.0 ~ 1.0)                   │
│   - 未完成原因分析                            │
│   - 已产生副作用说明                          │
│   - 后续建议                                 │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────┐
│ 第 4 层: 报告生成                            │
│ - 格式化输出                                 │
│ - 按用户偏好调整详略                          │
└─────────────────────────────────────────────┘
```

## 四、LLM 综合评估 Prompt 设计

```
Prompt 结构:

  系统提示:
    "你是 YellowBull 的结果评估员。
     请根据执行数据和成功标准，客观评估任务完成情况。"

  用户输入组装:

    【任务目标】
    {goal}

    【成功标准】
    {success_criteria}

    【执行统计】
    总步骤: {total}
    已完成: {done}
    已失败: {failed}
    已跳过: {skipped}
    完成率: {rate}%
    终止原因: {termination_reason}

    【关键步骤状态】
    {critical_steps_status_list}

    【失败步骤详情】
    {failed_steps_details}

    【子任务记录】
    {subtask_summary}

    【耗时统计】
    总耗时: {total_duration}
    各阶段耗时: {phase_durations}

    【约束违反情况】
    {constraint_violations}

    请返回:
    1. 最终结论: success / partial_success / failure
    2. 达成度: 0.0 ~ 1.0
    3. 未完成原因分析（如适用）
    4. 已产生的副作用说明（如适用）
    5. 后续建议（如适用）
```

## 五、报告生成

### 5.1 报告结构

```
┌─────────────────────────────────────────────┐
│ 报告头部                                     │
│ - 任务目标                                   │
│ - 最终结论（带达成度评分）                    │
│ - 总耗时                                     │
├─────────────────────────────────────────────┤
│ 执行摘要（一句话总结）                        │
├─────────────────────────────────────────────┤
│ 完成详情                                     │
│ - 完成的步骤列表（简要）                      │
│ - 关键成果                                   │
├─────────────────────────────────────────────┤
│ 问题说明（如有）                             │
│ - 失败的步骤及原因                            │
│ - 跳过的步骤及原因                            │
│ - 障碍排除记录                               │
├─────────────────────────────────────────────┤
│ 副作用说明（如有）                           │
│ - 文件变更                                   │
│ - 配置修改                                   │
│ - 依赖安装                                   │
├─────────────────────────────────────────────┤
│ 后续建议（如有）                             │
│ - 未完成的后续工作                            │
│ - 需要用户手动处理的事项                      │
└─────────────────────────────────────────────┘
```

### 5.2 报告详略级别

```
根据场景自动调整:

  级别 1 — 简洁模式
    触发条件: 完全成功 + 无副作用 + 用户偏好简洁
    内容: 头部 + 摘要（1-2 句话）

  级别 2 — 标准模式
    触发条件: 默认
    内容: 头部 + 摘要 + 完成详情 + 问题说明

  级别 3 — 详细模式
    触发条件: 部分成功/失败 + 有子任务 + 有副作用
    内容: 全部章节 + 子任务详情 + 每步耗时

  级别 4 — 调试模式
    触发条件: 用户配置或任务异常终止
    内容: 级别 3 + 原始执行日志 + context_store 快照

  默认展示策略: 摘要 + 可展开详情
    - 默认输出摘要级别
    - 详细结果折叠，用户按需展开
    - 提供"只看问题" / "只看成果" 过滤
```

### 5.3 报告格式

```
输出格式: 结构化 Markdown 文本

  示例:

  ## 任务: 检查所有 .py 文件的 import

  **结论:** 部分成功 (达成度 0.85)
  **耗时:** 3 分 22 秒

  ---

  ### 执行摘要
  检查了 25 个 Python 文件中的 23 个，发现 3 个文件存在 import 问题。

  ### 完成详情
  - 已检查: 23 / 25 文件
  - 通过: 20 个文件
  - 存在问题: 3 个文件

  ### 问题说明
  <details>
  <summary>展开查看失败详情</summary>

  | 文件 | 问题 | 原因 |
  |------|------|------|
  | utils.py | 检查失败 | 缺少 mypy 依赖（已自动安装） |
  | config.py | 检查失败 | 文件编码异常 |
  | test_helper.py | 已跳过 | 权限不足 |

  </details>

  ### 副作用
  - 安装了 mypy 依赖（可逆: pip uninstall mypy）

  ### 后续建议
  - 修复 config.py 的编码问题
  - 检查 test_helper.py 的权限设置
```

## 六、边界问题

### 6.1 任务异常终止

| 场景 | 处理方式 |
|------|---------|
| 用户中途取消 | 结论"已取消"，列出已完成步骤，提供"继续执行"选项 |
| 超时终止 | 判断已完成部分是否"够用"，够用→部分成功，不够→失败 |
| 预算耗尽 | 同超时终止，原因改为"步骤数超限" |
| 系统异常（崩溃） | MVP 不处理，后续版本支持断点恢复 |
| 用户中途修改目标 | 原任务标记 interrupted，新任务独立处理 |

### 6.2 结果数据缺失

| 场景 | 处理方式 |
|------|---------|
| done 但 context_store 无结果 | 视为"静默成功"（如删除操作），注明"执行成功，无输出" |
| failed 但有部分结果 | 保留部分结果，同时展示失败原因 |
| 子任务结果汇总到父任务 | 子任务成功→父步骤 done，失败→父步骤 failed，记录排除尝试 |

### 6.3 副作用处理

```
副作用检测方式: 工具声明 + LLM 补充

  1. 工具声明副作用
     每个工具声明可能的副作用类型:
       FileWrite, FileDelete, ConfigChange, DependencyInstall
     执行引擎记录每次副作用

  2. LLM 补充判断
     评估时额外判断:
       "根据执行结果，是否产生了未声明的副作用？"

  3. 报告中注明
     - 副作用类型
     - 可逆性: "可逆" / "不可逆" / "需手动回滚"
     - 可逆的提供回滚建议
```

### 6.4 用户不满意

```
任务标记成功，但用户说"不对":
  1. 记录用户反馈
  2. LLM 分析: "用户不满意的原因是什么？"
  3. 给出选项:
     a. 重新执行（完全重新拆解）
     b. 补充执行（只做缺失部分）
     c. 修正结果（手动修复）
     d. 放弃

任务标记失败，但用户说"其实已经够了":
  1. 记录用户反馈
  2. 更新达成度评分
  3. 将此次经验标记为"可接受的部分成功"
  4. 未来类似任务降低成功标准

用户对子任务结果有异议:
  1. 展示子任务详情
  2. 允许针对子任务单独反馈
```

### 6.5 循环/分支结果汇总

| 场景 | 处理方式 |
|------|---------|
| 循环执行了 50 次 | 不逐条列出，汇总统计 + 详细结果可展开 |
| 分支走了 false 路径 | 报告中注明分支决策，可展开查看 |
| 循环中触发子任务 | 报告中关联展示: "在 X 处遇到障碍 → 子任务 → 结果" |

### 6.6 数据一致性

| 场景 | 处理方式 |
|------|---------|
| step_status 与 context_store 不一致 | 按最保守方式处理，报告中注明 |
| 子任务写入父任务 context_store | 不允许，检测到→记录警告，不影响报告 |
| 报告生成本身失败 | 降级为机械统计报告，不经过 LLM 评估 |

### 6.7 结果数据量过大

```
- 循环 100 次迭代，每步大量输出
  → 报告中只展示汇总统计 + 前 N 条 + 最后 N 条
  → 完整数据可查询但不默认展示

- 报告生成长度限制
  → 超过 4096 字自动折叠详细部分
  → 保留摘要 + 统计
```

### 6.8 跨任务结果引用

```
- 结果按 task_id 索引，支持跨任务引用
- "上次那个任务的结果还能用吗？"
  → 查找历史任务，展示相关结果
- 需考虑结果过期策略（后续版本）
```

### 6.9 多目标任务

```
- "既要查 bug 又要优化性能"
  → 拆解时已拆成子目标
  → 报告按子目标分别评估
  → 给出总体结论 + 分项结论

示例:
  总体结论: 部分成功 (达成度 0.7)
  分项:
    - 查 bug: 成功 (达成度 0.9)
    - 优化性能: 失败 (达成度 0.5)
```

### 6.10 结果脱敏

```
- 工具输出可能包含密码、密钥、token
- 报告生成前做脱敏处理
- 规则: 匹配常见敏感模式
  - password=, token:, api_key, secret
  - 正则: (?i)(password|token|secret|key|api_key)\s*[=:]\s*\S+
- 脱敏后显示: [REDACTED]
```

### 6.11 零步骤任务

```
- 用户问"今天几号？"，拆解后只有 1 步
- 仍然走正常报告流程，只是极简
- 报告: "任务完成。今天是 2026 年 6 月 9 日。"
```

### 6.12 用户反馈收集

```
- 报告输出后，主动询问用户满意度
- 方式: 简单评分（满意 / 一般 / 不满意）
- 影响经验评分
- 用户未响应 → 默认"一般"
```

## 七、经验记录

### 7.1 自动记录

```
任务结束后自动提取经验:

  1. 提取维度
     - 任务类型: 代码修改 / 文件操作 / 分析诊断 ...
     - 工具链: 使用了哪些工具，顺序如何
     - 成功模式: 哪些步骤组合有效
     - 失败模式: 哪些步骤经常失败
     - 障碍模式: 哪些障碍反复出现

  2. 经验评分
     - 完全成功 + 无子任务 → +0.8
     - 完全成功 + 有子任务 → +0.5
     - 部分成功 → +0.2
     - 失败 → -0.3
     - 用户满意 → +0.2 加成
     - 用户不满意 → -0.2 惩罚

  3. 经验存储
     - 任务类型 → 工具链映射
     - 常见障碍 → 排除方案映射
     - 成功标准 → 实际达成度校准
```

### 7.2 经验 → 技能演化

```
经验积累到一定程度自动提炼为技能:

  1. 单次任务结束 → 记录一条经验
     {
       task_type: "代码分析",
       tool_chain: [FileRead, CodeAnalysis, ReportGenerate],
       success_rate: 0.9,
       obstacles: ["缺少依赖"],
       timestamp: "2026-06-09",
       user_satisfaction: "满意"
     }

  2. 同类经验不断积累
     - 代码分析经验 × 3
     - 代码分析经验 × 5
     - ...

  3. 达到阈值 → 触发技能提炼
     - 阈值: 同类经验 N 次（默认 5 次）
     - 条件: 成功率稳定 > 0.7
     - 工具链模式一致

  4. LLM 提炼技能
     "以下 5 次任务经验具有相似模式:
      - 工具链一致: FileRead → CodeAnalysis → ReportGenerate
      - 成功率: 平均 0.85
      - 常见障碍: 缺少依赖
      请提炼为可复用技能:
      - 技能名称
      - 适用场景
      - 标准步骤
      - 预检清单
      - 常见障碍及排除方案"

  5. 技能入库
     - 新技能经过用户确认
     - 加入技能库
     - 未来任务拆解时优先匹配

  6. 技能退化
     - 技能被使用后若连续失败
     - 权重下降
     - 降至阈值以下 → 降级为普通经验
```

## 八、结果持久化

### 8.1 需要保存的数据

```
  1. 任务执行记录（必存）
     - task_id
     - goal
     - 最终结论
     - 达成度
     - 执行时间
     - 步骤摘要

  2. 详细执行日志（可选）
     - 每步输入/输出
     - 子任务记录
     - 用户交互记录

  3. 经验数据（必存）
     - 任务类型标签
     - 工具链
     - 评分
```

### 8.2 存储策略

```
  - 摘要: 必存
  - 详细日志: 用户配置
  - 经验数据: 必存，永久保留
  - 结果数据: 保留到形成技能后，原始数据可归档

  经验保留策略:
    - 经验永久保留
    - 积累到阈值自动提炼为技能
    - 技能经用户确认后入库
    - 原始经验归档但不删除
```

## 九、重试机制

### 9.1 任务失败后重试

```
用户提供两种重试方式:

  方式 A — 完全重新拆解
    - 从头开始
    - 根据失败经验调整拆解策略
    - 可能走不同路径

  方式 B — 仅重试失败步骤
    - 保留已完成步骤
    - 只重新执行 failed 步骤
    - 快速恢复

  选择逻辑:
    - 用户未明确指定 → 自动判断
      失败步骤少（<30%）→ 方式 B
      失败步骤多（>=30%）→ 方式 A
    - 用户明确指定 → 按用户选择
```

### 9.2 重试时的经验利用

```
  - 查看失败原因
  - 调整工具选择
  - 调整步骤顺序
  - 预判可能障碍
  - 调整成功标准（如用户之前表示"其实够了"）
```

## 十一、MVP 阶段决策

| 设计点 | 决策 |
|-------|------|
| 报告默认展示 | 摘要 + 可展开详情 |
| 报告输出格式 | 结构化 Markdown 文本 |
| 经验记录时机 | 自动立即记录，不等待用户反馈 |
| 副作用检测 | 工具声明 + LLM 补充判断 |
| 经验保留策略 | 永久保留，积累后提炼为技能 |
| 重试方式 | 提供完全重新拆解和仅重试失败步骤两种方式 |
| 结果脱敏 | 匹配常见敏感模式 |
| 快照对比副作用 | MVP 不实现，后续版本 |
| 断点恢复 | MVP 不实现，后续版本 |
| 跨任务结果引用 | MVP 仅支持当前任务，后续版本支持跨任务 |
